当前实验执行链路：骨架 vs 缺口

绿色 = 已实现，黄色 = 有但不足，红色 = 缺失

整体流程：从文献到实验到报告

✅ 文献→报告链路（P1-P14，已完成）
PaperTriage → 筛选论文
↓
PaperReader → 提取证据
↓
SynthesisAgent → 综合报告
↓
ReviewerAgent → 审核
⚠️ 实验执行链路（骨架，无法闭环）
ExperimentPlanner → 生成实验计划（✅ 有规则+LLM）
↓
DeveloperAgent → 只创建 CodeTask，不改代码
↓
AutonomousExperiment → 只跑 smoke，不 debug
↓
ExperimentDecision → 分析结果，不反馈修复

四个关键缺口

🔴 缺口 1 Code Writer DeveloperAgent 只输出 CodeTask schema，不实际修改代码文件。没有 diff/patch 生成能力
🔴 缺口 2 Auto-Debug 循环 AutonomousExperiment 遇到错误就停止，不分析错误原因，不尝试修复。需要 feedback loop
🟡 缺口 3 结果分析深化 ExperimentDecision 只输出方向性建议 (investigate/continue/rollback)，不解析具体数值，不判断是否达到实验目标
🔴 缺口 4 闭环迭代 失败后无自动重规划。实验→分析→修复→再实验 的循环不存在

开源参考项目的解决思路

A

AI Scientist-v2 模式

实验树搜索 + 自动代码生成 + auto-debug（最多3轮修复）+ 结果驱动分支扩展。核心是 代码生成 agent 闭环

B

OpenHands CodeAct 模式

通用代码 agent（read/write/edit/bash），通过 sandbox 安全执行。核心是 "agent 在沙箱里操作" 的安全模式

C

Agent Laboratory 模式

阶段式流水线（文献→实验→报告），每阶段输出标准化 artifact。核心是 阶段间合约清晰、可替换

D

SWE-agent 模式

LM + 专用终端命令（search/edit/submit），通过 ACI（agent-computer interface）简化 agent 操作。核心是 简化 agent 的操作原语
